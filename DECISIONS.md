# Architecture Decisions — Forge3D

## D1 — Python + numpy + Pillow only
Justification: must run offline in the sandbox VM (no internet, no pip). This is the exact
available stack. No trimesh, no OpenCASCADE, no bpy.

## D2 — Write GLB from scratch
We implement the glTF 2.0 binary writer directly (see `gltf.py`). This gives us full
control over buffer packing, accessor types, and alignment, and avoids any dependency.

## D3 — Software rasterizer for PNG previews
We use numpy + matplotlib (both available) for a simple orthographic multi-view render.
Not photorealistic, but sufficient for QA and documentation.

## D4 — Degrees for all angles
Consistent with how artists and LLMs think. Internally converted to radians where needed.

## D5 — +Y up, metres
Matches glTF 2.0 spec (Y-up, metric). All primitives are centred at origin by default.

## D6 — One Mesh = one Material
Simplest possible data model. Multiple-material objects are represented as multiple
Mesh instances grouped in a Scene. This avoids multi-material UV bookkeeping.

## D7 — Indexed geometry everywhere
All export paths use indexed triangle lists (accessor type SCALAR, UNSIGNED_SHORT or
UNSIGNED_INT). Reduces GLB size 3-4x vs non-indexed.

## D8 — Texture resolution is a parameter (default 256)
Allows fast builds for examples; can be raised to 1024 or 2048 for production.

## D9 — Primitives centred at origin
box, sphere, cylinder, cone, torus, plane, capsule: bounding box is centred at (0,0,0).
This matches how artists expect primitives to behave and simplifies composition.

## D10 — boolean style — via CSG-lite mesh assembly
Full boolean (BSP) is complex and slow in pure Python. We instead use smart assembly:
for typical objects (subtractive holes, slots), we model the outer shell directly with
lathe/extrude/loft and mask with UVs. Full booleans are approximated via vertex welding.

## D11 — Bevel as UV-safe inset ring
Rounding/bevel on primitives is implemented as an extra ring of vertices near each edge
rather than true geometric bevel. This keeps the topology clean and UV-compatible.

## D12 — No Draco compression in GLB
Draco requires a decoder in the viewer. Our geometry is already small due to indexing.
Kept out to keep the viewer dependency-free (pure three.js, no wasm decoder).

## D13 — HDRI is a procedural EXP2 sky
Generated by `render.py::generate_studio_hdri()` as a 256x128 float RGBE file.
This gives subtle environment reflections without requiring a downloaded HDR.

## D14 — Spaceship wings as box+scale, not loft
Loft with 3-point rings produced degenerate normals (coplanar faces). Box primitives
scaled/positioned give correct normals and cleaner topology.

## D15 — bench COMPLEX budget = 5 s
Textured assets (wood, marble, leather) involve PIL image generation at 256x256 which
takes 1-2 s. 5 s is realistic for complex multi-mesh textured assets.
bench SIMPLE budget = 2 s (untextured metal/glass coins, mugs).

## D16 — Tests use stdlib unittest, not pytest
pytest is not installed in the VM and cannot be installed (no internet). All tests use
the standard `unittest` module which ships with Python.

## D17 — viewer uses ES module importmap + vendored three.js addons
Addon files (OrbitControls, GLTFLoader, HDRLoader, etc.) are vendored directly from
mrdoob/three.js via the GitHub API and committed to viewer/vendor/. The importmap
in viewer/index.html maps `three` to vendor/three.module.min.js.

## D18 — CI matrix: Python 3.11 + 3.12
Python 3.13 is used locally (VM) but GitHub Actions uses 3.11/3.12 which are the most
widely deployed. Both are tested to ensure broad compatibility.

## D19 — MIT license
Permissive, compatible with three.js (MIT) and numpy/Pillow (BSD/PIL).

## D20 — PYTHONPATH=src, no editable install
Avoids `pip install -e .` which requires internet. Users simply set PYTHONPATH=src.
